# 与 MySQL 审计的一致性说明（Phase 6-2）

本页聚焦当前实现中的一句话：  
**Redis 在接口入口挡重复，MQ 承担异步投递，MySQL 审计用于留痕对账；三者是组合策略，但不是单事务强一致。**

---

## 1. 先看当前真实调用顺序

根据现有 `MessageController` 与 `IdempotencyService`，一次首次请求的关键顺序是：

1. `markIfFirstTime(idempotencyKey)`：Redis `SET NX EX` 抢占幂等键
2. `sendMessage(payload)`：发送到 `demo.queue.boot`
3. `saveAuditRecord(idempotencyKey, payload)`：写 `idempotency_record`

重复请求会在第 1 步直接返回“重复请求，已忽略”，后续 2/3 不执行。

---

## 2. 一致性边界：为什么它不是强一致

以上 1/2/3 分别落在 **Redis / MQ / MySQL**，不是同一个本地事务资源。  
这意味着：系统是“**最终一致 + 可补偿**”思路，而非“单次调用绝对原子”。

典型窗口：

- **窗口 A**：2 成功、3 失败  
  - 现象：消息已入队并可能被消费，但审计表缺这条记录。
- **窗口 B**：1 成功、2 失败  
  - 现象：幂等键已占用，但消息未入队；短期内同 key 重试会被挡住（直到 TTL 到期或人工处理）。
- **窗口 C**：2 成功后应用异常退出  
  - 现象：调用方已收到失败/超时感知，但后台可能已处理消息（“客户端视角失败，系统侧可能成功”）。

这些都不是 bug，而是跨系统调用天然存在的分布式一致性问题，需要制度化补偿。

---

## 3. 当前方案的价值与代价

### 价值

- **高并发入口挡重**：Redis `SET NX EX` 成本低、速度快。
- **可审计**：MySQL 表可对账、追踪请求键与 payload。
- **与 DLQ 联动**：失败消息可进死信，支持后续人工处置与回放。

### 代价

- **写库与发消息之间有窗口**：需要接受“短时不一致”。
- **幂等键 TTL 是业务策略**：TTL 太短会放过重复，TTL 太长会挡住合法重试。
- **回放必须幂等**：否则可能二次副作用。

---

## 4. 企业级常见补偿手段（从轻到重）

1. **审计状态机（推荐先做）**
   - 在 `idempotency_record` 增加状态：`INIT / SENT / CONSUMED / FAILED / DLQ`
   - 每个关键事件更新状态，便于排查“卡在哪一步”。

2. **定时对账任务**
   - 扫描“Redis 已占位但 DB 无记录”或“SENT 长时间未消费”的记录
   - 生成告警或补偿工单。

3. **Outbox 模式（更强一致）**
   - 先在同一 DB 事务写业务数据 + outbox 事件，再由异步发布器发 MQ
   - 可显著缩小“发消息/写审计”不一致窗口，复杂度也更高。

---

## 5. 本项目的落地建议（不改架构前提）

短期（当前 Demo 阶段）：

- 保持现有流程，明确文档语义为“**最终一致**”
- 增加最少量可观测信息：请求日志里串上 `idempotencyKey`
- 对 `saveAuditRecord` 失败加明显错误日志与报警入口

中期（下一阶段）：

- 扩展审计表状态字段并补一条“异常补偿说明”
- 与下一章“消息回放”衔接：以审计记录为回放入口，重放前再次校验幂等键

---

## 6. 面试/评审可直接复用的表述

> 我们采用 Redis + MQ + MySQL 审计的组合：Redis 负责入口快速幂等，MQ 负责异步解耦，MySQL 负责审计留痕。  
> 这套方案吞吐高、落地快，但它不是单事务强一致。我们通过审计状态、DLQ、告警和回放补偿来实现最终一致。
